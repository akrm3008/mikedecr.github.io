A Love Letter to list2env()

A quick note on reproducibility and remote computing

Michael DeCrescenzo
Sep 27, 2019 6 min read Methods

Motivation

This is a short post about a function that I didn’t know about until recently: list2env(). In brief, list2env() does the following:

  • takes a named list as an argument
  • “multiply assigns” list components to objects with respective names
  • You can pick which environment these objects are assigned to. This can be the global environment if you want.

An example of how it works in action:

# create a named list
my_list <- list(a = 1, b = 2)

# assign list objects
list2env(my_list, envir = .GlobalEnv)
## <environment: R_GlobalEnv>
# did objects make it into environment?
ls(); a; b
## [1] "a"       "b"       "my_list"
## [1] 1
## [1] 2

So that’s what list2env() does, but why would you use it? I will describe a workflow that I have been implementing for my dissertation.

  • I do a lot of work with Bayesian models/MCMC. I estimate these models on a remote machine (connected using ssh) so not to overburden my laptop.
  • Objects created on the remote machine (posterior samples, the number of Markov chains, number of iterations, etc.) have to get passed back to my machine.
  • Once on my machine, read data into R in a format that is convenient. I then use these objects for analysis or for writing an .Rmd document.

Where list2env() comes in

Why is this function special? Mainly because of what won’t work for me. This remote-local hybrid workflow requires a few things to go right.

The whole routine needs to be reproducible. I’m a huge geek about making sure my projects run from top to bottom. But more importantly in the near term, I have to do this routine a ton, because a lot of my work requires me to bounce between multiple machines. I can’t just email a one-off data file to myself. This has to work with code, and it has to work smoothly.

When I’m on the remote machine, I don’t want to save the whole workspace. I don’t need to keep every object that I create on the remote computer. I only want to use some of these later. So while a routine that involves saving the whole R workspace as an .Rdata might work when you’re on a local machine without too many objects, I want to be careful which things I save from the remote machine.

Slightly longer demo

To see it in practice, pretend we’re on the remote machine. We have created a handful of objects, only some of which we want to save. I will add some objects to our environment to demonstrate.

data(iris)
data(mtcars)

Let’s see what all we have now.

ls()
## [1] "a"       "b"       "iris"    "mtcars"  "my_list"

Now, suppose I want to pass data back to my home machine, but I only want to keep the objects b and iris. I could do this by putting them into a list. These objects should be named!

objs_to_save <- list(b = b, iris = iris)

Now imagine that I upload the files to Box. This is done using boxr::box_write(objs_to_save, "saved-list.RDS", dir_id = ...) where ... is a numeric ID assigned to a synced folder on Box.com. (If I wanted to save the whole .Rdata image, I could use box_save_image()).

Now pretend I’m on my home machine again. I will wipe our workspace (all but objs_to_save) to simulate the effect.

rm(list = ls()[ls() != "objs_to_save"])

I read the saved list from Box, so now it’s in my workspace as a list:

ls()
## [1] "objs_to_save"
lapply(objs_to_save, head)
## $b
## [1] 2
## 
## $iris
##   Sepal.Length Sepal.Width Petal.Length Petal.Width Species
## 1          5.1         3.5          1.4         0.2  setosa
## 2          4.9         3.0          1.4         0.2  setosa
## 3          4.7         3.2          1.3         0.2  setosa
## 4          4.6         3.1          1.5         0.2  setosa
## 5          5.0         3.6          1.4         0.2  setosa
## 6          5.4         3.9          1.7         0.4  setosa

Here is where we use list2env(). We simply drop the objects in this list into the global environment.

list2env(objs_to_save, envir = .GlobalEnv)
## <environment: R_GlobalEnv>
ls()
## [1] "b"            "iris"         "objs_to_save"

Now we can use these objects for analysis in an R script, or we can use them in an .Rmd report to write up the results of what we did on the remote machine. Whatever you’re doing, saving objects as a list and then reading the list into R can be a quick way to move data between R sessions, and list2env() makes it easy.
